# M4.5 — Dashboard local (FastAPI + HTMX + Chart.js)

Spec d'implémentation du Module 4.5 de polycopy : dashboard web **local** et **read-only** pour visualiser en quasi-temps-réel les trades détectés, les décisions strategy, les ordres executor, les positions ouvertes et le PnL. **Aucune action d'écriture**, **aucune API publique exposée**, **bind `127.0.0.1` exclusif par défaut**. Pas de remplacement de `scripts/pnl_report.py` — le dashboard vit à côté pour la supervision live.

Source de vérité fonctionnelle : `docs/architecture.md` (à compléter §10.2) + cette spec. Conventions : `CLAUDE.md`. Schémas DB : `src/polycopy/storage/models.py` (inchangé). Templates : `specs/M1-watcher-storage.md` → `specs/M4-monitoring.md` (mêmes sections 0..15, mêmes critères d'acceptation en checklist).

## 0. Pré-requis environnement

### 0.1 Bootstrap (déjà fait)

`bash scripts/setup.sh` (idempotent). Aucun patch config structurel requis — M4.5 n'ajoute que 3 env vars optionnelles + 3 deps. `scripts/setup.sh` devra lancer `pip install -e ".[dev]"` pour tirer les nouvelles deps (rien à patcher, `pip install -e` est déjà idempotent).

### 0.2 Nouvelles dépendances

Dans `pyproject.toml` `[project] dependencies` :

- `fastapi>=0.110.0` — framework web async.
- `uvicorn[standard]>=0.27.0` — serveur ASGI, deps optionnelles pour `websockets`/`httptools` (gain perf négligeable mais `[standard]` est la recommandation FastAPI officielle).
- `jinja2>=3.1.3` — templates HTML server-side.

**Note** : pas de frontend build tool (pas de node, pas de Vite, pas de Tailwind). HTMX + Chart.js + Pico.css servis depuis CDN (arbitré §2.5). Pas de dep dev supplémentaire — `httpx` est déjà présent (utilisé pour les tests ASGI `AsyncClient + ASGITransport`).

### 0.3 `.env` — nouvelles variables (toutes OPTIONNELLES)

| Variable env | Champ Settings | Default | Description |
|---|---|---|---|
| `DASHBOARD_ENABLED` | `dashboard_enabled` | `false` | Opt-in strict. Si `false`, `__main__` n'instancie **pas** le `DashboardOrchestrator` (zéro overhead, zéro port ouvert). |
| `DASHBOARD_HOST` | `dashboard_host` | `127.0.0.1` | Bind explicite localhost-only. Changer à ses risques (cf. §2.2). |
| `DASHBOARD_PORT` | `dashboard_port` | `8787` | Port TCP local. Arbitré §2.3 — non conflictuel avec les ports dev classiques (3000/5000/8000/8080). |

À ajouter à `config.py` ET `.env.example` (avec commentaires d'avertissement sécurité pour `DASHBOARD_HOST`). Aucune n'est requise — les 3 defaults marchent et le dashboard reste **off par défaut**.

### 0.4 Critère de validation "environnement"

```bash
DASHBOARD_ENABLED=true python -m polycopy --dry-run
```

Doit logger en plus des M1/M2/M3/M4 events :

- `dashboard_starting` (binding : `host=127.0.0.1`, `port=8787`).
- `dashboard_started`.
- `dashboard_stopped` au shutdown.

Un `curl http://127.0.0.1:8787/` retourne 200 avec du HTML (page Home). Un `curl http://<wsl-ip>:8787/` (ou `curl http://0.0.0.0:8787/` depuis une autre machine) doit **échouer** (connexion refusée) — preuve du bind localhost-only.

Exit 0 sur SIGINT (le serveur uvicorn stoppe via `server.should_exit = True`).

### 0.5 Sécurité — rappels stricts pour M4.5

- **JAMAIS** bind `0.0.0.0` par défaut. L'utilisateur qui override `DASHBOARD_HOST=0.0.0.0` accepte que son dashboard soit joignable sur toutes les interfaces réseau (WSL peut router vers le LAN Windows) → risques : lecture de ses détections wallets publiques + expose qu'il fait du copy trading. Aucune creds ne fuite (cf. point suivant), mais la donnée DB reste sensible.
- **AUCUN leak** de `TELEGRAM_BOT_TOKEN`, `POLYMARKET_PRIVATE_KEY`, `POLYMARKET_FUNDER`, ou creds CLOB L2 (`api_key`, `api_secret`, `api_passphrase`) dans les réponses HTML, JSON ou logs d'accès. Vérifié par un test automatisé (§9.7).
- **Aucun endpoint write**. Pas de `POST /orders/cancel`, pas de `POST /dry-run/toggle`, pas de `DELETE /positions/<id>`. Toutes les routes FastAPI sont `GET`. Le dashboard ne peut littéralement pas modifier la DB (toutes les sessions sont `async with session_factory() as s: await s.execute(select(...))` — zéro `add`, `commit` write, ou `delete`).
- **Pas de secret en clair côté client**. Zéro JS inline qui referencerait une clé. HTMX + Chart.js uniquement — pas d'auth côté navigateur.

## 1. Objectif M4.5 (scope exact)

Donner au bot une **interface de supervision live** locale :

- 1 page Home avec KPIs (total_usdc, drawdown, positions ouvertes, trades/ordres 24h, dernière alerte).
- 5 pages de drill-down : Détection (trades), Stratégie (décisions), Exécution (ordres + fills), Positions (ouvertes + fermées), PnL (graph Chart.js sur fenêtre configurable).
- Rafraîchissement 2–5 s via HTMX polling (`hx-trigger="every 3s"`).
- Lancement automatique quand `DASHBOARD_ENABLED=true`, co-orchestré avec watcher/strategy/executor/monitoring dans le même `asyncio.TaskGroup` du `__main__`.

**Livrable fonctionnel** : `DASHBOARD_ENABLED=true python -m polycopy --dry-run` ouvre un serveur uvicorn in-process sur `127.0.0.1:8787`. L'utilisateur ouvre un navigateur sur `http://127.0.0.1:8787/` et voit en live ses détections, décisions, ordres et PnL.

**Hors livrable M4.5** : auth (bind localhost suffit), actions write, WebSocket/SSE (polling HTMX), export CSV (cf. `pnl_report.py`), PWA, i18n, leaderboard traders (M5).

## 2. Arbitrages techniques (7 points à trancher explicitement)

### 2.1 Process model — uvicorn in-process dans le `TaskGroup` existant

**Recommandation : in-process TaskGroup.**

Pros :
- Cohérence totale avec `MonitoringOrchestrator` (M4) qui lance `PnlSnapshotWriter` + `AlertDispatcher` dans un `TaskGroup` interne. On ajoute juste `DashboardOrchestrator.run_forever(stop_event)`.
- Un seul `stop_event` partagé → shutdown atomique sur SIGINT (uvicorn reçoit `server.should_exit = True` via un wrapper async).
- `session_factory` et `settings` partagés directement par injection — zéro sérialisation, zéro process IPC.
- Un seul process à monitorer côté utilisateur (`ps -ef | grep polycopy`).

Cons :
- Crash uvicorn → crash bot (mais le TaskGroup protège : `except* Exception` peut isoler). En pratique uvicorn est très stable en local.
- Partage de GIL → mais le dashboard sert ~3 req/s max (1 user), négligeable.

Alternative rejetée — **subprocess** : ajoute un second process à gérer, nécessite IPC ou lecture DB concurrente, double le risque de zombies, zéro bénéfice à 1 user local.

Alternative rejetée — **script séparé** : demande à l'utilisateur de `python -m polycopy.dashboard` dans un 2e terminal. Moins ergonomique, et le dashboard devient un outil externe au lieu d'une feature intégrée.

### 2.2 Auth — aucune (bind localhost suffit)

**Recommandation : aucune auth applicative.**

Pros :
- Bind `127.0.0.1` = le dashboard n'est joignable que depuis la machine elle-même. Aucun attaquant réseau ne peut y accéder sans compromettre déjà le host.
- Zéro complexité (pas de password storage, pas de session cookie, pas de JWT rotation).
- Cohérent avec le philosophie "bot solo personnel, pas un produit" du README.

Cons :
- Un user multi-tenant (ex: shared dev box) verrait le dashboard d'un autre user → hors scope, le bot est mono-utilisateur par design.
- Si l'utilisateur override `DASHBOARD_HOST=0.0.0.0` sans réaliser la conséquence → documenté dans §0.3 table et dans `.env.example` (`# ⚠️ changer à vos risques`). Pas de garde-fou code (l'utilisateur est adulte).

Alternative rejetée — **Basic HTTP** : ajoute un header `Authorization: Basic` à stocker en env var, une comparaison constant-time, et expose un token en base64 dans les logs navigateur/uvicorn — mauvaise hygiène pour un bénéfice nul en bind localhost.

Alternative rejetée — **JWT** : over-engineering absolu pour 1 user.

### 2.3 Real-time — HTMX polling `hx-trigger="every 3s"`

**Recommandation : HTMX polling.**

Pros :
- Zéro code serveur async-streaming — chaque requête est un round-trip GET classique.
- Simplicité : `<div hx-get="/partials/kpis" hx-trigger="every 3s" hx-swap="innerHTML">`. C'est tout.
- Chart.js re-render gratuit : le swap HTMX remplace le `<canvas>` → Chart.js démarré dans un `htmx:afterSwap` listener.
- Debug trivial (on voit les requêtes dans DevTools Network).

Cons :
- ~0.3 req/s/panel si 3s d'intervalle × 5 panels visibles → ~1.5 req/s → négligeable (SQLite en local soutient 10k req/s).

Alternative rejetée — **SSE (`EventSource`)** : nécessite une tâche async de fan-out côté serveur, gère les reconnexions, supporte mal les proxies. Gain inexistant pour 1 user.

Alternative rejetée — **WebSocket** : idem SSE + complexité client. Hors scope (§13).

Alternative rejetée — **Chart.js auto-refresh** : non-natif, devrait être polyfillé en JS custom.

### 2.4 Frontend CSS — Pico.css classless via CDN

**Recommandation : Pico.css v2 classless, CDN.**

Pros :
- ~10 KB gzipped, dark mode natif via `<html data-theme="dark">`.
- **Classless** : on écrit du HTML sémantique pur (`<article>`, `<nav>`, `<table>`), zéro markup-soup de classes utility.
- Typo system-ui par défaut, cohérent avec `docs/architecture.md`.
- Zéro build step, zéro node_modules.

Cons :
- CDN requiert internet au premier chargement. Mitigé : le navigateur cache agressivement, et M4.5 est un outil local de supervision (l'utilisateur est déjà online pour lire ses détections).

Alternative rejetée — **Tailwind** : nécessite un build step ou un CDN jit de 300 KB. Hors proportion pour 6 pages de tables + 1 graph.

Alternative rejetée — **CSS custom minimal** : réinvente la roue pour ~100 lignes vs 0 ligne avec Pico.

Alternative rejetée — **htmx-components** : niche, lock-in à un framework naissant.

### 2.5 Chart.js — CDN à M4.5, vendoring reporté

**Recommandation : CDN `https://cdn.jsdelivr.net/npm/chart.js@4.4.x`.**

Pros :
- Zéro gestion de binaire vendored dans le repo (~200 KB).
- Version pinned (major.minor), upgrade contrôlé.
- Cache agressif navigateur entre sessions.

Cons :
- Pas d'offline. Si besoin offline futur (cas edge : dashboard dans un VPS isolé), vendorer dans `src/polycopy/dashboard/static/vendor/chart.min.js` en M5+ (2 lignes de diff, changement de `<script src>`).

Même logique pour HTMX (`https://cdn.jsdelivr.net/npm/htmx.org@2.x`) et Pico.css (`https://cdn.jsdelivr.net/npm/@picocss/pico@2/css/pico.min.css`).

### 2.6 Structure des routes — multi-page avec partials HTMX

**Recommandation : URL par page (`/`, `/detections`, `/strategy`, `/orders`, `/positions`, `/pnl`) + sous-URLs `/partials/*` pour les fragments HTMX.**

Pros :
- Le navigateur gère l'historique (back/forward fonctionnent).
- Chaque page est autosuffisante (refresh fonctionne).
- Les partials HTMX (`/partials/kpis`, `/partials/orders-rows?status=FILLED`) sont de simples templates Jinja inclus — pas de logique supplémentaire.
- Debug facile : l'URL dans la barre d'adresse décrit l'état.

Cons :
- Un peu plus de routes (~12 total : 6 pages + 6 partials). Acceptable pour la clarté.

Alternative rejetée — **SPA avec swap global** : casse le back/forward browser, nécessite un router JS, déborde du scope "pas de frontend build".

### 2.7 Logs — structlog seulement, `uvicorn.access` désactivé

**Recommandation : `log_config=None` à uvicorn.Server + désactiver `uvicorn.access` via un log level élevé.**

Pros :
- Uniformité avec le reste du bot (logs JSON structlog partout).
- Pas de duplication `uvicorn.access` + `dashboard_request`.
- Logs exploitables par le rapport M4+ futur.

Cons :
- Perte du format uvicorn `INFO:     127.0.0.1:XXXXX - "GET / HTTP/1.1" 200 OK`. Remplacé par `log.info("dashboard_request", method="GET", path="/", status=200, duration_ms=3)` via un middleware ASGI minimaliste.

Uvicorn config : `uvicorn.Config(app, host=..., port=..., log_config=None, access_log=False)`. Le serveur propre vivra dans `src/polycopy/dashboard/server.py`.

## 4. Lifecycle uvicorn — `src/polycopy/dashboard/server.py`

### 4.1 Problème : stopper uvicorn depuis `stop_event`

`uvicorn.Server.serve()` est une coroutine qui tourne indéfiniment. Pour la stopper proprement sur `stop_event.set()`, il faut set `server.should_exit = True` depuis une tâche watchdog.

### 4.2 Pattern retenu

```python
async def serve_dashboard(
    app: FastAPI,
    host: str,
    port: int,
    stop_event: asyncio.Event,
) -> None:
    """Sert `app` via uvicorn jusqu'à `stop_event.set()`."""
    config = uvicorn.Config(
        app,
        host=host,
        port=port,
        log_config=None,
        access_log=False,
        lifespan="on",
    )
    server = uvicorn.Server(config)

    async def _watchdog() -> None:
        await stop_event.wait()
        server.should_exit = True

    async with asyncio.TaskGroup() as tg:
        tg.create_task(server.serve())
        tg.create_task(_watchdog())
```

- `server.serve()` sort proprement quand `should_exit=True` (drain des connections ouvertes, puis retour).
- Le `TaskGroup` interne garantit que si l'un des deux tasks crashe, l'autre est annulé.

### 4.3 Bind explicite

`uvicorn.Config(host=settings.dashboard_host, ...)`. Si l'utilisateur laisse le default `127.0.0.1`, le kernel refuse tout `SYN` qui n'arrive pas via `lo` (loopback). Testable via `ss -tlnp | grep 8787` → `127.0.0.1:8787` et pas `0.0.0.0:8787` ni `*:8787`.

## 5. Routes FastAPI — `src/polycopy/dashboard/routes.py`

### 5.1 Injection de `session_factory` via `Depends`

```python
def get_session_factory(request: Request) -> async_sessionmaker[AsyncSession]:
    """Retourne le ``session_factory`` attaché à l'app FastAPI."""
    factory: async_sessionmaker[AsyncSession] = request.app.state.session_factory
    return factory
```

`DashboardOrchestrator` (§8) fait `app.state.session_factory = self._session_factory` au démarrage. Toutes les routes utilisent `session_factory: Annotated[async_sessionmaker[AsyncSession], Depends(get_session_factory)]`. Pas de middleware custom, pas de ContextVar.

### 5.3 Pagination simple

`offset`/`limit` query params, default `offset=0`, `limit=50`, `limit` max 200 (clamp côté serveur). Footer template : `<a href="?offset={{ offset + limit }}">next</a>`. Pas de count total (évite un `SELECT COUNT(*)` à chaque page, non critique en M4.5).

### 5.4 Filtres

- `?wallet=0xabc` (case-insensitive côté SQL) sur `/partials/detections-rows`.
- `?decision=APPROVED` ou `REJECTED` sur `/partials/strategy-rows`.
- `?status=FILLED|SENT|SIMULATED|REJECTED|FAILED|PARTIALLY_FILLED` sur `/partials/orders-rows`.
- `?state=open|closed` sur `/partials/positions-rows`.
- Valeurs invalides → ignorées silencieusement (pas de 422, UX > strictness).

### 6.2 Note sur "dernière alerte Telegram"

Les `Alert` M4 **ne sont pas persistées** en DB (cf. `specs/M4-monitoring.md` §4.2 : "Pas d'écriture DB des alertes à M4 (uniquement logs structlog + envoi Telegram)"). Donc `HomeKpis.last_alert_*` seront **toujours `None` à M4.5**. On affiche "—" dans le template, et on ajoute un `# TODO M5: persister les alertes dans une table pour historisation dashboard` dans `queries.py`. **Ne pas refactorer M4** dans ce PR.

### 6.3 Fenêtres temporelles PnL

`?since=1h` → `timedelta(hours=1)`, `24h`, `7d`, `30d`. Valeur par défaut `24h`. Parsing helper `_parse_since(s: str) -> timedelta` dans `queries.py`, défensif (fallback à `24h` si invalide).

### 7.2 HTMX polling

Exemple Home :

```html
<section id="kpis" hx-get="/partials/kpis" hx-trigger="load, every 3s" hx-swap="innerHTML">
  <p aria-busy="true">chargement KPIs…</p>
```

### 7.3 Chart.js + HTMX

Chart.js n'est pas re-créé à chaque swap (coûteux). Pattern :
- Le template `pnl.html` contient un `<canvas id="pnlChart">` + un `<div>` invisible polling `/partials/pnl-data.json` toutes les 5 s.
- Un petit script inline (~30 lignes dans `pnl.html`) écoute `htmx:afterSwap` sur ce div, parse le JSON, et appelle `chart.data.labels = ... ; chart.update()`.
- Alternative plus simple : un `<script>` au chargement qui `setInterval(fetch("/partials/pnl-data.json"), 5000)`. **Retenue** car plus lisible (pas de magie HTMX pour du JSON).

### 7.4 Markup soigné

- Tableaux avec `<thead>`/`<tbody>`, `role="grid"` pour Pico.
- Dates rendues en ISO UTC compact (`2026-04-18 12:34:56Z`).

## 8. `DashboardOrchestrator` + intégration `__main__`

### 8.1 `DashboardOrchestrator` — `src/polycopy/dashboard/orchestrator.py`

```python
class DashboardOrchestrator:
    def __init__(
        self,
        session_factory: async_sessionmaker[AsyncSession],
        settings: Settings,
    ) -> None:
        self._session_factory = session_factory
        self._settings = settings

    async def run_forever(self, stop_event: asyncio.Event) -> None:
        log.info(
            "dashboard_starting",
            host=self._settings.dashboard_host,
            port=self._settings.dashboard_port,
        )
        app = build_app(self._session_factory, self._settings)
        try:
            await serve_dashboard(
                app,
                host=self._settings.dashboard_host,
                port=self._settings.dashboard_port,
                stop_event=stop_event,
            )
        finally:
            log.info("dashboard_stopped")
```

`build_app(session_factory, settings) -> FastAPI` vit dans `routes.py`. Il :

1. Crée `FastAPI(docs_url=None, redoc_url=None, openapi_url=None)` — **pas de Swagger UI en prod** (zéro endpoint public).
2. `app.state.session_factory = session_factory`.
3. `app.state.settings = settings` (si une route a besoin d'un flag de config — ex: afficher "DRY-RUN" dans le header).
4. Monte le router `pages_router` + `partials_router` + `StaticFiles("src/polycopy/dashboard/static")`.
5. Ajoute le middleware `StructlogAccessMiddleware` (§2.7).
6. Configure `Jinja2Templates(directory="src/polycopy/dashboard/templates")`.
7. Log `dashboard_started` (binding : routes_count).

### 8.2 Modifications `__main__._run()`

Ajouts **guard par `settings.dashboard_enabled`** :

```python
from polycopy.dashboard.orchestrator import DashboardOrchestrator  # import lazy OK
```

## 14. Notes d'implémentation / décisions arbitrées

**Ordre de travail suggéré** :

1. Ajouter les 3 deps (`fastapi`, `uvicorn[standard]`, `jinja2`) dans `pyproject.toml`, `pip install -e ".[dev]"`.
2. Ajouter les 3 env vars dans `config.py` + `.env.example` (avec commentaires sécurité pour `DASHBOARD_HOST`).
3. Créer l'arbo `src/polycopy/dashboard/` (§3).
4. `queries.py` + `test_dashboard_queries.py` (TDD — queries pures, faciles à tester).
5. `routes.py` (pages + partials + `get_session_factory` dep) + `test_dashboard_routes.py` via ASGI `AsyncClient`.
6. Templates Jinja (`base.html`, macros, 6 pages, 6 partials) — minimalisme Pico.css.
7. `middleware.py` (access log structlog) + `server.py` (wrapper uvicorn async avec watchdog stop_event).
8. `test_dashboard_server.py` (lifecycle).
9. `test_dashboard_security.py` (no-leak secrets, no-write, no-docs) — **bloquant avant merge**.
10. `orchestrator.py` + `test_dashboard_orchestrator.py`.
11. Intégration `__main__` (guard `if settings.dashboard_enabled`).
12. Smoke test manuel : `DASHBOARD_ENABLED=true python -m polycopy --dry-run`, ouvrir chaque page, vérifier polling HTMX en DevTools.
13. Docs §10 (README + architecture + CLAUDE + setup §13).
14. Commit unique.

**Principes** :

- **Pas d'abstraction prématurée** : 1 `queries.py` concret, pas de `AbstractDashboardReader`. Les templates Jinja consomment directement les objets ORM (pas de sérialisation Pydantic inutile).
- **Sessions courtes, zéro lock long** : 1 session par fonction `queries.*`, fermée avant rendering. Jinja n'a jamais de session ouverte. SQLite en async mode support 1 writer + N readers — on ne crée que des readers.
- **Logs structurés** : `dashboard_starting`, `dashboard_started`, `dashboard_stopped`, `dashboard_request` (method, path, status, duration_ms), `dashboard_request_error` (avec traceback sur 500). Pas d'`access_log` uvicorn natif.
- **Pas de `print`, pas de JS inline massif** (juste ~30 lignes dans `pnl.html` pour le fetch JSON + `chart.update()`).
- **Import FastAPI/uvicorn au module level OK** — cost ~50 ms au boot, négligeable. Pas d'import lazy.
- **`app.state.session_factory`** : meilleur choix pour FastAPI + asyncio (vs ContextVar ou globals). Testable, inspectable, injectable via `Depends`.

**Décisions tranchées en auto-arbitrage (pour éviter les allers-retours)** :

1. **Logo / branding dashboard** : sobre, un simple `<strong>polycopy</strong>` dans le `<nav>`. Pas d'ASCII art, pas d'emoji.
2. **Favicon** : un tiny SVG `data:image/svg+xml` embedé dans `base.html` (lettre P dans un cercle). Pas de fichier .ico à vendored.
3. **Date/time rendering** : ISO 8601 UTC compact (`2026-04-18 12:34:56Z`). Pas de "3 minutes ago" relatif (évite Intl.RelativeTimeFormat, pénible à tester). L'utilisateur avancé lit de l'UTC.
4. **Erreur 500 côté route** : middleware `StructlogAccessMiddleware` catch, log `dashboard_request_error`, return `500 Internal Server Error` HTML minimal (Pico template `error.html`). Pas de leak de stacktrace côté user (mais conservée dans les logs).
5. **Chart.js data format** : endpoint `/partials/pnl-data.json` retourne `{"timestamps": [ISO strings], "total_usdc": [floats], "drawdown_pct": [floats]}`. Chart.js reçoit `labels = timestamps.map(new Date)` et 2 datasets (line chart avec 2 axes Y : USDC à gauche, drawdown % à droite).
6. **Home KPIs** : pas de sparkline SVG en Home (on a un tab PnL dédié avec Chart.js). Juste des "big numbers" via macros `kpi_card`.
7. **Paramètre `since` du Chart.js** : géré côté client via des `<button hx-get="/pnl?since=24h">` qui swap toute la page `/pnl` (pas juste le canvas) — plus simple que de re-créer Chart.js en JS.

**Pièges anticipés** :

1. **Uvicorn + TaskGroup** : `uvicorn.Server.serve()` gère sa propre boucle signal handling. Désactiver via `config.lifespan="on"` et laisser `__main__` gérer les signaux (déjà en place). Si conflit → `config.install_signal_handlers = False` (méthode privée historique ; fallback si besoin).
2. **FastAPI + async sessions SQLAlchemy** : ne **jamais** passer un `AsyncSession` via `Depends` direct — ça crée un lifecycle compliqué. Passer le `session_factory` et créer la session dans la route (`async with session_factory() as s:`).
3. **HTMX + `hx-swap="innerHTML"`** : si le fragment contient un `<script>`, il faut `hx-swap="innerHTML" hx-swap-oob="true"` pour l'exécuter. À éviter — mettre les scripts en `base.html` et les données en attributs `data-*`.
4. **CDN down** : si jsdelivr est down, dashboard cassé. Mitigation : fail gracefully (la page reste lisible sans JS, Pico.css manque → fallback `<link>` data URI inline minimal si vraiment requis à M5+). À M4.5 on accepte la dép CDN.
5. **WSL IP vs `localhost`** : sous WSL2, `127.0.0.1` dans le bot est joignable depuis Windows via `http://localhost:8787/` grâce au port forwarding auto de WSL2. Documenter dans `docs/setup.md` §13 si l'utilisateur se plaint.
6. **Ports conflict** : 8787 est peu utilisé (ni node, ni django, ni FastAPI default). Si conflit rare → override `DASHBOARD_PORT`.
7. **Jinja2 auto-escape** : FastAPI `Jinja2Templates` active auto-escape par défaut. OK — on veut escape systématique des valeurs DB (slugs utilisateurs, labels trader). Ne pas désactiver.
8. **`app.state` typing** : FastAPI `Request.app.state` est `Any`. Accéder via un helper `get_session_factory(request)` avec annotation explicite (`async_sessionmaker[AsyncSession]`) pour mypy.
9. **Test ASGI + lifespan** : `ASGITransport(app=app, lifespan="off")` si on n'a pas besoin des startup/shutdown events (cas courant). Sinon wrapper `LifespanManager` de `asgi-lifespan` (mais overkill — FastAPI n'a pas de lifespan critique en M4.5).
10. **Migration de port** : si l'user avait un autre service sur 8787, il override via `DASHBOARD_PORT`. Aucun impact sur le reste du bot.
